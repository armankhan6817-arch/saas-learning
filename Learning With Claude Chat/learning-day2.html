Dungeon 1 finale + the sticky-note model + full Dungeon 2 · pre-build for the to-do app
How to revise with this file: same as day 1 — say the output out loud before tapping "Reveal answer". Purple cards are concept deep-dives from my own counter-questions; those are the highest-value re-reads.

⚔️ The rules earned today

  • The sticky-note model: variables never refer to each other — each holds its own copy of an address. No wire between notes; only from each note to the object.
  • The global = logic: = never travels. The left-hand expression does. Bare variable on the left → rewrites that note. Dot/bracket on the left → travels through the address, modifies the shared thing.
  • Closure: an inner function keeps access to its outer function's variables even after the outer returns — it carries a backpack 🎒. Each call to the outer function stitches a fresh backpack.
  • Callback: a function you hand over (no ()), to be called later by someone else — the browser, map, filter. You provide the what; they decide the when.
  • Definition parens vs calling parens: parens between function and { = parameter list. Parens after a function value = play the card.
  • Strings are immutable. Every string method returns a new string; the original is never touched.
  • Missing let inside a function = grabbing (and possibly hijacking) the outer variable.

Level 4 · The Summoning Circle — closuressolved with hints ✓
function makeCounter() {
  let count = 0;

  return function() {
    count = count + 1;
    return count;
  };
}

let counterA = makeCounter();
let counterB = makeCounter();

console.log(counterA());
console.log(counterA());
console.log(counterB());
Locals die when their function returns… so how is count surviving?
Reveal answer

1, 2, 1

makeCounter() returns a function — an unexecuted recipe card. counterA catches it (Q1 lesson: return values must be caught). The () in counterA() plays the card.

The riddle: normally count dies at return — but the inner function still uses it, and JS keeps a variable alive as long as some function needs it. The returned function carries its birthplace's variables in a backpack 🎒.

counterB = makeCounter() is a separate call → fresh count = 0, fresh backpack → logs 1, not 3. Two counters, two independent counts.

Depth calibration: recognition level only for now. Skim javascript.info 6.3 first half later; skip lexical-environment theory. Why it's non-negotiable: react hooks ARE closures — useState only makes sense with the backpack idea.
Level 5 · Final Boss — The Amalgamtwo attempts, then won
function process(team) {
  team.push("dave");
  team = ["eve"];
  team.push("frank");
  return team;
}

let heroes = ["alice", "bob"];
let result = process(heroes);

console.log(heroes);
console.log(result);
Which lines touch the original array, which don't, and why?
Reveal answer

heroes: ["alice", "bob", "dave"] · result: ["eve", "frank"]

Line by line:

1. team.push("dave") — mutation through the shared address → heroes' array gains "dave".
2. team = ["eve"] — reassignment → team's note now points at a brand-new array. Link to heroes broken.
3. team.push("frank") — mutates the new array only → ["eve", "frank"]. Push adds to the end.
4. return team — returns the new array; call site catches it into result.

My mistakes: first swapped the effects entirely; second attempt had "frank, eve" — forgot push appends to the end, not the front.
What this boss integrates: Q1 (call site catching) + Q4 (shared reference) + Q5 (mutate vs reassign) in one snippet — the full week-2 reference model.
Part 2 · The Sticky-Note Model (my counter-question)
Deep-dive · why doesn't team = ["eve"] touch heroes?concept
My question: "whenever we use team it refers to heroes… so why didn't team = ["eve"] add eve to heroes?"
Reveal the model

Correction first: team never refers to heroes. Both are independent variables holding the same address. No wire between the sticky notes — only from each note to the array.

heroes → [ 📦 #742 ] ─┐ ├──→ #742: ["alice","bob"] team → [ 📦 #742 ] ─┘ after team = ["eve"]: heroes → [ 📦 #742 ] ──→ #742: ["alice","bob","dave"] team → [ 📦 #901 ] ──→ #901: ["eve"]

team.push(...) → reads team's note, travels to #742, modifies the array there. Heroes' note also says #742, so it sees the change.

team = ["eve"] → never visits #742. Builds a new array at #901, overwrites team's note only. Assignment only ever changes the variable on the left of =.

The global logic (no exceptions): = means "compute the right side, store it in the slot named on the left." The only question: where does that slot live?

• team — slot is the variable itself. No travel.
• team[0], team.leader, team.settings.theme — slot lives inside the object. JS must resolve team, travel to the address, then write into the slot there.

= never travels. The left-hand expression travels — or doesn't. And .push() is the travel case in disguise: resolving team.push requires the trip.
Refinement of my first paraphrase: it's not the = sign that decides — it's what's on its left. team = ["eve"] (bare) = reassignment. team[0] = "eve" (bracket) = mutation. Both have =.
The hijack · missing letfirst try ✓
function process(team) {
  heroes = team;      // no let!
  team.push("x");
}

let heroes = ["alice"];
let villains = ["zed"];

process(villains);
console.log(heroes);
What logs? And what changes if it were let heroes = team?
Reveal answer

["zed", "x"] — no let means the line grabs the outer heroes and rewrites its note to villains' address. The push then mutates that shared array. Both variables now show ["zed","x"]; poor ["alice"] is abandoned.

With let: a brand-new local variable shadows the outer one, exists for two lines, dies at function end. Outer heroes untouched → logs ["alice"]. (The push still mutates villains either way.)

Bug class unlocked: accidentally grabbing outer variables by forgetting let. Same shadowing behavior in a parameter (Q1), a loop block (Q3), or any { }.

Level 2 · Reducefirst try, full trace ✓
let cart = [120, 80, 250];

let total = cart.reduce((sum, price) => sum + price, 0);

console.log(total);
Reveal answer

450

pass 1: sum=0, price=120 → 120 pass 2: sum=120, price=80 → 200 pass 3: sum=200, price=250 → 450

Each pass's return value becomes the next pass's sum; the trailing 0 seeds pass 1. That running accumulator is all reduce is.

Recognition only: the accumulator can be an object or array too — e.g. turning a todo list into counts per category. Read it when you meet it.

Deep-dive · Callbacks (my counter-question)concept
"What do you mean by callback, and why didn't function() {...} run immediately — its parens are right there?"
Reveal the model

Callback = a function you hand to someone else, to be called later, by them, not you.

greet;     // the card itself. nothing runs.
greet();   // () AFTER a function value plays the card.

The parens confusion: in function() {...}, those parens are the parameter list — anatomy of the definition (function name(params) { body }), here just empty. Position is everything: parens between function and { = parameter list. Parens after a finished function value = calling parens.

Run it immediately? Wrap and call — (function(){...})() — an IIFE. Arrows make it visible: () => {...} is the definition; (() => {...})() defines and plays on the spot. Recognition only.

I've used callbacks all along: map(fn) — once per item · filter(fn) — per item, keep if true · reduce(fn, 0) — per pass · addEventListener("click", fn) — per click, forever. You provide the what; the other function decides the when and how-many-times.

Classic trap: addEventListener("click", greet()) — the () plays the card NOW; greet returns nothing → the browser is handed undefined as its "card". Q1 + Q2 ghosts reunited. Correct: pass greet bare.
React preview: onClick={handleAdd} — note: no parens. Same rule, week 5.
Final Boss · The To-Do Golem3/3 first strike ✓✓
let tasks = [
  { text: "learn js", done: true },
  { text: "build app", done: false },
  { text: "ship it", done: false }
];

let pending = tasks.filter(t => !t.done);
let labels = pending.map(t => t.text.toUpperCase());

let { text } = tasks[1];

let summary = `${pending.length} left: ${labels.join(", ")}`;

console.log(labels);
console.log(text);
console.log(summary);
Reveal answer

["BUILD APP", "SHIP IT"]
"build app"
"2 left: BUILD APP, SHIP IT"

!t.done = "not done" → filter keeps two tasks · map uppercases into a new array · destructuring grabs text from tasks[1] · template literal + .join(", ") assembles the summary.

Log 2 stays lowercase because toUpperCase returned new strings — immutability, applied.
