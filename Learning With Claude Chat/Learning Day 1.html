Day 1 — Scope, Return Values & References

8 questions + 3 dungeon levels · javascript.info ch. 2–6 territory
How to revise with this file: read each snippet, say the output out loud before tapping "Reveal answer". If you get one wrong, close the file and rewrite that snippet from memory tomorrow. That's the active-recall rule.

⚔️ The rules you earned today

  • Primitives copy the value. Objects copy the address.
  • A return value must be caught at the call site (x = fn(x)) or it's discarded.
  • The whole expression fn(x) becomes the returned value — not the argument inside it.
  • Mutate (push, obj.prop = ...) → original changes. Reassign (list = ...) → original untouched.
  • No return → function gives undefined (never null — humans write null, JS gives undefined).
  • let lives only inside its { }. var leaks out. A for-loop's ( ) counts as inside.
  • Spread [...arr] = shallow copy: new shelf, same books.
Part 1 · Scope & Return Values
Q1 · parameter shadowingmissed, then won
let count = 5;

function increase(count) {
  count = count + 1;
  return count;
}

increase(count);
console.log(count);
What logs, and why?
Reveal answer

Logs 5.

Two separate things happen: (1) the parameter count is a copy — incrementing it never touches the outer variable. (2) the returned 6 reaches the call site, but nothing catches it — no = — so it's discarded.

My mistake: answered 6, thought the return "went back into" the outer count. It doesn't — return hands the value to the call site; the call site must catch it.

Fix: count = increase(count); → logs 6.

Shop analogy: the shop hands back your change — you still have to take it.
Q2 · missing returnfirst try ✓
function double(x) {
  x * 2;
}

let a = double(4);
console.log(a);
What logs?
Reveal answer

Logs undefined.

x * 2 computes 8 and throws it away — there's no return. A function without return gives undefined automatically.

undefined vs null: JS hands you undefined ("nothing here"). null only appears when a human deliberately writes it.

Mirror of Q1: Q1 returned but didn't catch. Q2 computed but didn't return. Either way, the value is lost.
Q3 · block scope + loop countermissed, then won
let message = "start";

for (let i = 0; i < 3; i++) {
  let message = "loop " + i;
}

console.log(message);
console.log(i);
What does each line print? (One might not print at all…)
Reveal answer

Line 1: "start" — the inner let message is a brand-new variable that shadows the outer one inside the { }, then dies.

Line 2: ReferenceError: i is not defined — crash. let i in the loop's ( ) belongs to the loop's block scope. It dies when the loop ends.

My mistake: answered "loop 2" and "2" — forgot that let inside braces creates a separate, temporary variable.

Follow-up I got right: with var i instead, no crash — var ignores block scope and leaks out. It prints 3, not 2: the loop exits when i++ pushes i to 3 and i < 3 fails. The counter always ends one past the last iteration.

Part 2 · References — the big divide
Q4 · array referencefirst try ✓
let nums = [10, 20, 30];
let copy = nums;

copy.push(40);

console.log(nums.length);
What logs — what did copy = nums actually copy?
Reveal answer

Logs 4.

copy = nums copies the reference (the address), not the array. One array, two names. Pushing through either name changes the same shared array.

The rule: primitives (numbers, strings, booleans) copy by value. Objects and arrays copy by address.

Independent copy: let copy = [...nums] (spread) or structuredClone(nums) for deep copies.

Q5 · mutate vs reassignhalf right, then won
function replace(list) {
  list = ["new"];
}

function modify(list) {
  list.push("new");
}

let arr1 = ["old"];
let arr2 = ["old"];

replace(arr1);
modify(arr2);

console.log(arr1);
console.log(arr2);
What does each log?
Reveal answer

arr1: ["old"] — list = ["new"] is a reassignment: it repoints the local parameter at a new array. The original is never touched.

arr2: ["old", "new"] — push is a mutation: it goes through the shared reference and changes the original.

My mistake: said arr1 would be "new". Reassigning a parameter is Q1 all over again — just with an address instead of a number.
My breakthrough: I thought return put the value "into the argument". Actually the whole expression replace(arr1) becomes the returned value — and it still needs arr1 = to catch it. Also: when you do catch it, the old array is abandoned (nothing points at it), not modified — a second variable still pointing at it would still see ["old"].
Q6 · mapfirst try ✓
let prices = [100, 250, 50];

let doubled = prices.map(p => p * 2);

console.log(prices[0]);
console.log(doubled[1]);
Does map mutate? What is p => p * 2 doing?
Reveal answer

100 and 500.

map never mutates — it builds a new array by running the arrow function on each item: [200, 500, 100].

p => p * 2 = "take p, return p*2" — implicit return, no keyword needed in the short form.

File away: mutators: push, pop, sort, splice. New-array makers: map, filter, slice. When unsure → MDN.

Q7 · filter + map chainfirst try ✓
let scores = [45, 90, 72, 30, 88];

let result = scores
  .filter(s => s >= 70)
  .map(s => s + 10);

console.log(result);
console.log(scores.length);
Trace it in two stages.
Reveal answer

Stage 1 — filter keeps items passing the test: [90, 72, 88].

Stage 2 — map adds 10 to each survivor: [100, 82, 98].

scores.length is still 5 — original untouched.

Why it matters: "filter the todos, map them to components" is the shape of half the code in any React app.
Part 3 · The Scope Dungeon 🎮
Level 1 (Q8) · object vs primitive extractionhalf right, then won
let user = {
  name: "armaan",
  streak: 4
};

let profile = user;
profile.streak = profile.streak + 1;

console.log(user.streak);

let name = user.name;
name = "claude";

console.log(user.name);
Why does the first change leak through but the second doesn't?
Reveal answer

5 then "armaan".

profile = user copies the address → mutation through profile hits the shared object.

user.name is a string (primitive) → name gets an independent copy. Reassigning it never touches the object.

My mistake: said "claude" — forgot that extracting a primitive out of an object gives a safe copy.
The rule: grab an object → you hold a live wire. Grab a primitive out of it → you hold a photocopy.
Level 2 · off-by-oneclean kill ✓
let items = ["sword", "shield", "potion"];

for (let i = 0; i <= items.length; i++) {
  console.log(items[i]);
}
What prints — all lines — and what's the bug?
Reveal answer

sword · shield · potion · undefined

The bug: off-by-one. <= lets i = 3 through, and items[3] doesn't exist → JS answers undefined (no crash, no null — same ghost as Q2).

Fix: i < items.length.

Contractor note: AI writes this bug too. Now you'll spot it instantly.
Level 3 · shallow copyhalf right, then won
let party = [
  { name: "knight", hp: 100 },
  { name: "mage", hp: 60 }
];

let squad = [...party];

squad[0].hp = 25;
squad.push({ name: "rogue", hp: 80 });

console.log(party[0].hp);
console.log(party.length);
What did spread actually copy — the objects, or something sneakier?
Reveal answer

party[0].hp: 25 — spread made a new array, but the items inside are object references. squad[0] and party[0] are two names for one knight — stab him through either, he bleeds in both.

party.length: 2 — the push stayed contained in the new array.

My mistake: said hp would be 100 — forgot spread is shallow: new shelf, same books.

Deep copy (photocopies the books too): structuredClone(party).

Why it matters: React state updates lean on spread constantly. "Why did my other component's data change??" is almost always this trap.
